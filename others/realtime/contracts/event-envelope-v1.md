# Realtime Event Envelope v1

## 1. Purpose

Define the canonical event envelope for all socket messages between client and server.

## 2. Envelope Schema

```json
{
  "event": "string",
  "version": "1.0",
  "requestId": "uuid-string",
  "timestamp": 1712736000000,
  "data": {},
  "error": {
    "code": "string",
    "message": "string",
    "details": {}
  }
}
```

## 3. Field Rules

1. event
   - Required.
   - Must be one of registered names in event-catalog-v1.md.
2. version
   - Required.
   - Must be string and default to 1.0.
3. requestId
   - Required.
   - Must be UUID-like string.
   - Echoed in ack or nack.
4. timestamp
   - Required.
   - Unix epoch milliseconds generated by sender.
5. data
   - Optional for events with no payload.
   - Must follow event-catalog payload schema.
6. error
   - Present only for nack or server error events.
   - code must exist in error-codes-v1.md.

## 4. Ack and Nack Envelope

Ack:

```json
{
  "event": "ack",
  "version": "1.0",
  "requestId": "f4a6d4a5-11f7-4a9d-8f3f-1cf0304d7a65",
  "timestamp": 1712736001000,
  "data": {
    "sourceEvent": "message.send"
  }
}
```

Nack:

```json
{
  "event": "nack",
  "version": "1.0",
  "requestId": "f4a6d4a5-11f7-4a9d-8f3f-1cf0304d7a65",
  "timestamp": 1712736001000,
  "error": {
    "code": "AUTH_REQUIRED",
    "message": "Authentication token is missing",
    "details": {
      "sourceEvent": "room.join"
    }
  }
}
```

## 5. Validation Order

Server validation order for inbound events:

1. Envelope shape validation.
2. event and version validation.
3. requestId validation.
4. Auth and permission validation.
5. Event-specific payload validation.

## 6. Backward Compatibility

1. v1 clients must ignore unknown fields.
2. Breaking field changes require new version and migration note.
3. Additive fields are allowed within v1 if marked optional.
